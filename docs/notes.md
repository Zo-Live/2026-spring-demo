# Implementation Notes

这份笔记记录了本项目从开始到完成过程中出现过的主要问题，以及最终采用的修复方案。按时间和模块整理，尽量覆盖整个实现过程中的失误、偏差和回退。

## 1. 初始搭建阶段

### 问题：仓库里残留了旧目录和无关文件
- 现象：根目录下有旧的 `gomoku/`、`micro-civ/`、`bomberman/`、`campus-scheduler/`、`compare-notes.md` 等内容。
- 影响：容易和新骨架混淆，后续实现时难以保证统一结构。
- 解决方案：
  - 按要求清理了除 `docs/`、`LICENSE`、`.git/`、`.gitignore` 外的旧内容。
  - `.codex` 因挂载占用无法删除，保留但不参与项目结构。

### 问题：直接用 `python` 运行命令失败
- 现象：环境里没有 `python` 命令，只有 `python3`。
- 影响：测试和 demo 运行命令全部报错。
- 解决方案：
  - 后续执行统一改用 `python3`。
  - 根 README 和各 demo README 中的命令也统一改为 `python3 ...`。

## 2. 统一交互层阶段

### 问题：最初的 `curses` 网格选择器只适合很短的标题
- 现象：`civilization`、`campus`、`bomberman` 后续把状态、规则、提示一起塞进标题区后，`addstr()` 超出终端边界，直接触发 `_curses.error: addwstr() returned ERR`。
- 影响：手动模式在进入建城/选格等环节时崩溃。
- 解决方案：
  - 给共享层增加了 `_safe_addstr()`，所有写屏操作都先裁剪宽度和高度。
  - `select_grid_curses()` 和 `select_menu_curses()` 改成支持多行标题。
  - 对部分界面再进一步做“紧凑标题头”，避免信息过多。

### 问题：网格选择器的装饰风格一开始过于死板
- 现象：默认逻辑是当前格用方括号、非法格用圆括号。
- 影响：
  - `gomoku` 和 `bomberman` 看起来很乱。
  - 后续用户要求不同 demo 的选中样式不一样时，不方便复用。
- 解决方案：
  - 在共享层给 `select_grid()` 增加了：
    - `decorate_unselectable`
    - `selected_style`
  - 让不同 demo 可以选择：
    - 当前格用方括号
    - 当前格用反色高亮
    - 非法格是否使用圆括号

## 3. Gomoku

### 问题：最初 baseline AI 过于基础
- 现象：AI 只会：
  - 己方立即成五
  - 挡住对手立即成五
  - 其余时候简单贴近已有棋子
- 影响：不会提早防守开放三/潜在长连，也没有进攻布局。
- 解决方案：
  - 重写了 [demos/gomoku/ai.py](/home/zolive/2026-spring-demo/demos/gomoku/ai.py)。
  - 引入：
    - 候选点限制在已有棋子邻域内
    - 连子长度评分
    - 开放端数量评分
    - 己方进攻分与对手威胁分综合比较
  - 新 AI 能更早防守开放三、开放四，并做基本中盘布局。

### 问题：显示符号前后不一致
- 现象：
  - 外部静态棋盘显示 `● / ○`
  - 手动选择界面仍显示 `B / W`
  - 当前玩家提示也仍显示 `B / W`
- 影响：视觉不统一。
- 解决方案：
  - 统一成：
    - 空格 `.`  
    - 黑棋 `●`
    - 白棋 `○`
  - 手动界面、自动界面、当前玩家提示、胜者提示都同步更新。

### 问题：高亮方式第一次实现错了
- 现象：改成“反色高亮”后，第一次实现会把整列一起高亮。
- 原因：共享网格渲染在所有行都按同一列做了反色处理，没有限制到当前行。
- 解决方案：
  - 只在 `row == y` 时做反色渲染。
  - 现在只高亮当前选中的单个格子。

### 问题：手动界面不该再给普通棋子加括号
- 现象：曾经使用 `[●]`、`(○)` 之类的装饰。
- 影响：比静态棋盘更难读。
- 解决方案：
  - 手动界面取消普通格子的括号装饰。
  - 当前光标使用反色而不是括号。

## 4. Micro Civilization

### 问题：最初手动模式几乎不可用
- 现象：玩家只能在“地图页”和“选项页”之间来回切换，做完动作后没有明确反馈。
- 原因：
  - 状态展示和输入选择分离成两个独立 screen。
  - 合法性过滤不足，导致很多操作选了以后才报错或没有结果感。
- 解决方案：
  - 手动模式重做：
    - 状态和操作放进同一套 screen
    - 建城时只允许选合法格
    - 建筑/科技菜单只显示合法选项
    - 每次动作结算后弹出结果页，显示本轮日志

### 问题：建城界面一进入就崩溃
- 现象：`build city` 直接触发 `_curses.error`
- 原因：
  - 标题太长
  - 地图宽度较大
  - `curses.addstr` 越界
- 解决方案：
  - 公共层增加安全裁剪
  - `civilization` 的建城界面改用紧凑标题头，只保留必要状态和提示

### 问题：地图对齐差
- 现象：
  - emoji 宽度混杂
  - 城市格使用 `🏙️0` 之类“emoji + 数字”，打乱列宽
  - 顶部列号和网格不对齐
- 解决方案：
  - 改成固定单元显示：
    - 普通地形统一包在 `| 🌲 |` 这种更宽单元内
    - 城市格只显示 `🏠`，不再在地图内显示编号
    - 编号仍放在下方城市列表中
  - 列号表头改成与列宽一致的居中布局
  - 先后修过两次：
    - 第一次错误地只在数字两边加了竖线
    - 第二次按要求去掉竖线，只保留居中数字并与列宽对齐

### 问题：建城选择页提示语和实际逻辑相反
- 现象：提示写的是“带方括号的可选”，但实际是当前光标方括号、非法格圆括号。
- 影响：误导用户。
- 解决方案：
  - 改成明确说明：
    - 先移动到合法格
    - 非法格会显示圆括号

### 问题：科技研究界面把 `education` 挤出可见区域
- 现象：一开始研究科技菜单标题太高，导致最下面的选项在某些终端里不可见。
- 解决方案：
  - 科技菜单使用超紧凑标题，只保留 `turn / science / score`
  - 让全部科技选项可以同时显示

### 问题：收益机制实现了，但说明不清楚
- 现象：虽然代码中已经有：
  - 地形基础收益
  - 城市 3x3 区域产出
  - 科技解锁建筑
  - 建筑增加每回合产出
  但界面里没有清楚说明。
- 解决方案：
  - 在 [demos/civilization/render.py](/home/zolive/2026-spring-demo/demos/civilization/render.py) 增加常驻说明面板，明确写出：
    - 城市收益计算方式
    - 地形对应资源
    - 科技的作用
    - 建筑的作用

## 5. Campus Scheduler

### 问题：最初手动模式没有明确状态反馈
- 现象：玩家选地点后感觉“没有发生任何事”。
- 原因：
  - 没有把每小时结算日志在手动模式中展示出来
  - 状态界面和操作界面分裂
- 解决方案：
  - 每次 `step()` 后弹出结果页
  - 展示最近几条日志
  - 让玩家知道是“开始移动”“到达地点”还是“执行动作”

### 问题：二维地图不适合这个题目
- 现象：地点本来就只有 5 个，但最初仍用 10x10 网格展示。
- 影响：信息密度低，且手动操作不直观。
- 解决方案：
  - 按要求改成竖列表展示地点
  - 手动模式直接在地点列表中选择目标地

### 问题：状态说明不足
- 现象：最初界面虽然有部分状态，但缺少玩法说明：
  - 不知道体力/饱食度机制
  - 不知道 Debuff 触发条件
  - 不知道建筑开放时间
  - 不知道成就条件
- 解决方案：
  - 在界面常驻显示：
    - 开放时间
    - `fatigue` / `hunger` 触发条件和效果
    - 成就解锁条件
  - 每个学生显示：
    - 精力
    - 饱食度
    - 学分
    - 健康
    - 心情
    - Debuff
    - 成就

### 问题：地点列表里显示 `pos=(x,y)` 没意义
- 现象：用户明确指出不需要二维坐标。
- 解决方案：
  - 地点列表去掉 `pos`
  - 仅保留地点名、emoji、容量与开放时间信息

### 问题：选项显示不全，看起来只剩两个地点可选
- 现象：手动模式里用户看到的只有 `dorm` 和 `teaching`。
- 原因：标题区过高，终端高度不够时后面的菜单项被裁掉。
- 解决方案：
  - 新增紧凑版 dashboard
  - 把标题压缩到数行内
  - 保证全部地点选项都可见

### 问题：玩家不知道各地点之间的移动时间
- 现象：虽然引擎里已经实现了 `distance_hours()`，但界面不展示。
- 解决方案：
  - 在当前学生选地点时，常驻显示：
    - 从当前位置到每个地点分别需要多少小时

### 问题：闭馆状态的可见性不足
- 现象：用户提到关门后仍可能继续停留在楼里，界面也不易判断当前地点是否开放。
- 当前解决方案：
  - 在当前所在地后显示 `🟢/🔴`
  - 执行动作时仍会检查开放时间
- 说明：
  - 当前版本没有实现“闭馆自动赶人/强制离场”
  - 这是有意保持阶段一最小规则闭环，未继续扩展

## 6. Bomberman

### 问题：最初单人模式体验错误
- 现象：玩家只控制 `P1`，但 AI `P2` 会开局自杀式埋雷，很快结束对局。
- 影响：看起来像“只能走几步就自动结束”。
- 原因：初版 baseline AI 太激进，只要靠近对手就放炸弹，没有检查退路。
- 解决方案：
  - 重写 [demos/bomberman/ai.py](/home/zolive/2026-spring-demo/demos/bomberman/ai.py)
  - 新策略：
    - 先避开即将爆炸区域
    - 只有确认自己能逃离当前炸弹 blast 时才埋雷
    - 其余时间优先走安全格

### 问题：怪物看起来不动
- 现象：用户观察时认为怪物不会动。
- 原因：
  - 最初怪物候选动作包含原地，很多局面里容易频繁留在原地
  - 棋盘显示又不够清晰
- 解决方案：
  - 修改怪物移动逻辑：
    - 有合法邻格时优先在邻格中随机
    - 只有完全无法移动时才原地
  - 同时提升了棋盘渲染可读性

### 问题：手动模式找不到放炸弹选项
- 现象：虽然逻辑上支持“选中当前格 -> bomb / wait”，但界面不够明显。
- 解决方案：
  - 明确在说明中写出：
    - 选邻格 = 移动
    - 选当前格 = 弹出 `Place bomb / Wait`
  - 若当前位置不能放炸弹，则菜单中不显示 `Place bomb`

### 问题：手动模式也会崩溃
- 现象：和 `civilization` 一样，在进入手动选格界面时触发 `_curses.error`
- 原因：同样是标题和棋盘组合过高/过宽导致越界写入。
- 解决方案：
  - 复用共享层的安全写屏修复
  - 同时把 `bomberman` 手动模式标题压缩成紧凑版

### 问题：地图对齐和括号样式不好看
- 现象：
  - emoji 和 `.` 宽度不一致
  - 非选中格也带装饰括号
- 解决方案：
  - 地图改成统一宽单元格
  - `.` 特殊按窄字符单独补空格
  - 非选中格取消括号
  - 当前格保留方括号作为选中反馈

## 7. README 与文档

### 问题：运行命令和环境说明最初不够准确
- 现象：最初写成了 `python -m ...`
- 解决方案：
  - 全部 README 改为 `python3 -m ...`

### 问题：阶段性说明分散
- 现象：一开始只有汇总 README，没有把各 demo 的限制写清楚。
- 解决方案：
  - 每个 demo 单独写 `README.md`
  - 根 README 负责统一入口、目录结构、测试方法

## 8. 测试阶段

### 问题：测试运行会生成 `__pycache__`
- 现象：测试后目录里出现缓存文件。
- 影响：仓库不整洁。
- 解决方案：
  - 在一次测试后专门清理了 `__pycache__`

### 问题：部分测试最初和实际规则不匹配
- 现象：
  - `campus` 的疲劳测试放在错误时间段，导致动作本身未执行
  - `bomberman` 的阻挡测试对初始地图假设不稳定
- 解决方案：
  - 调整测试输入，使测试更精确地覆盖目标规则

## 9. 当前状态总结

到最终版本为止，主要修复方向包括：
- 统一结构
- 强化 baseline AI
- 修复手动模式不可用
- 解决 `curses` 崩溃
- 改善地图与菜单渲染
- 为用户补齐规则说明

仍然保留的“有意简化”包括：
- `gomoku` 没有做高级搜索
- `civilization` 仍是最小经营版，不做复杂科技树/扩张代价
- `campus` 不做闭馆强制离场
- `bomberman` 不做道具、强化、复杂链式爆炸

这些保留项是为了维持“阶段一最小可运行 demo”的目标，而不是最终大作业级完成度。
